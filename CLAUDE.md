# CLAUDE.md

## Operating principles

- Before writing code, consult the relevant section of [SPEC.md](SPEC.md) (the WHAT) and [ARCHITECTURE.md](ARCHITECTURE.md) (the HOW).
- Run `cargo test` and `cargo clippy --all-targets` before declaring a change done.

## About

**revolver** is a Rust lightweight UPnP/DLNA MediaServer. Single binary, SQLite backend, LAN-only. Verified on Linn DSM/2 (real device).

## Commands

```sh
cargo build --release      # LTO + strip
cargo test
cargo clippy --all-targets
cargo fmt
```

Admin UI: `http://localhost:8200/` (scan / reshuffle / stats / settings / scan progress). `/admin/ui` and `/admin/` redirect here (308) for backward compat.

One-time per clone, enable the in-repo git hooks so `cargo fmt --check` + `cargo clippy -- -D warnings` run before every commit (same checks as CI):

```sh
git config core.hooksPath .githooks
```

## Key design decisions

Things you cannot grasp by reading one file:

- **Two-table model: `albums` + `tracks`** (SPEC Â§3). `album_id` is the primary key for UPnP ObjectIDs and all queries. Album identity = `(effective_album_artist, album, compilation)` â€” `album_id` stays stable across tag fixes as long as identity doesn't change.
- **`effective_album_artist` is computed at scan time** and stored: `compilation` â†’ "Various Artists"; else `album_artist` â†’ `artist` â†’ "Unknown Artist". Do not recompute in queries.
- **`albums.quality` is bulk-UPDATEd after scan** (SPEC Â§4.6). Tiers: `flac/alac/pcm` + (>48kHz or >16bit) â†’ `hires`; else `lossless` / `lossy` / `mixed` / `unknown`. Browse hits it via index.
- **ObjectID stability is the cross-client compatibility key** (SPEC Â§6.1, Â§10.4). `alb:{album_id}` / `trk:{track_id}` are auto-increment based and permanent. `aa:{base64}` / `ar:` / `gn:` are URL-safe base64 of the name (`-` `_`, no padding). Never break IDs on rescan.
- **`cat:recent` is a flat album list** (SPEC Â§6.7, #16). Sorted by `albums.last_added_at` DESC, capped by `browse.recently_added_limit` + optional `browse.recently_added_max_age_days`. The earlier time-range cascade (`day` / `week` / `year:YYYY` / `all`) was dropped after real-device use showed the two-click hop was friction without value. Future Views default to flat â€” avoid sub-container cascades.
- **SystemUpdateID increments only on structural change** (SPEC Â§5.1). `scan::should_bump_system_update_id` decides; bump on scan complete + fan-out propchange NOTIFY to CD subscribers. Playback and random reshuffle do NOT bump (avoid trashing Linn's Browse cache). Currently also bumps on `tracks_updated > 0` â€” slightly over-bumpy but harmless.
- **`added_at` = "when the server first saw this path"** (SPEC Â§4.2), not file btime. First scan: btime/mtime fallback. After: now(). Never overwrite on upsert. Recently Added orders by `MAX(added_at) by album_id`, so a new track on an existing album re-floats the album.
- **Skip tag read when mtime matches** (SPEC Â§4.5). Reported as `tracks_unchanged`. Makes "unchanged rescan" finish in seconds.
- **M4A: detect ALAC vs AAC via `lofty`** and store in `codec` (SPEC Â§14). Both are `audio/mp4` in protocolInfo, but `bitsPerSample` is emitted only for lossless.
- **Cross-client strategy: UPnP AV 1.0 standard for client-payload surfaces** (DIDL-Lite, SearchCriteria, ObjectIDs, container hierarchy â€” SPEC Â§10). Vendor extensions (Linn `X_MAP_*`, Sonos `r:`, etc.) are case-by-case per Â§10.2: must be verifiable on real hw + invisible to other clients + measurably improve UX. Currently none emitted. Recently Added / Played / Random / Quality use virtual containers (SortCriteria is unused by Linn UI â€” SPEC Â§6.7-6.8).
- **Random Albums is an in-memory `Mutex<Vec<i64>>`** (SPEC Â§6.6). Reshuffled at startup / scan complete / `POST /admin/reshuffle`. Not persisted â€” a fresh order each launch is fine. A **lazy time-based re-roll** layers on top: `browse.random_albums_shuffle_interval_hours` (default `Some(24)` â€” daily re-roll, since always-on LAN servers rarely restart; set to `null` via admin UI to freeze the order between events) makes [`RandomState::maybe_reshuffle`](src/random.rs) re-shuffle at Browse time when the configured interval has elapsed since the previous reshuffle. Lazy on purpose â€” no tokio timer task burns idle hours, and `last_shuffled_at` lives next to `album_ids` in the same struct so a Mutex pair is enough. Failures inside `maybe_reshuffle` are logged via `tracing::warn` and fall through to the previous order, so `cat:random` never serves an empty page because the lazy refresh stumbled. `browse.random_albums_limit` defaults to `Some(100)` so the out-of-the-box `cat:random` is a browsable page-sized random selection rather than a full library shuffle.
- **Recently Played counts stream hits** (SPEC Â§6.8). Only `Range` absent or `bytes=0-N` increments `play_count` + `last_played_at`. `bytes=N-` (N>0) and suffix `bytes=-N` are excluded (reject pre-fetch). Same rule for all clients â€” no Linn-specific heuristics.
- **Album art: on-demand extract + bytes-budget cache** (SPEC Â§8.3). Full clear past 100MB (no LRU â€” keep it simple).
- **Scan runs inside `tokio::task::spawn_blocking`** with rayon inside. Never blocks a tokio worker.
- **DB uses r2d2 connection pool** (rusqlite + spawn_blocking). SQLite WAL mode allows Browse during scan.
- **Schema migration: idempotent ALTER via `ensure_column`**. After `CREATE TABLE IF NOT EXISTS`, check columns via `PRAGMA table_info`. `CREATE INDEX IF NOT EXISTS idx_trk_played` must run AFTER `ensure_column` â€” the index fails if the target column doesn't exist on older DBs.
- **`BrowseContext` centralizes view-wide deps** (`conn` / `art_base_url` / `stream_base_url` / `random_state` / `now_secs`). The caller (`content_directory.rs`) injects the clock so tests are time-independent.
- **Shutdown broadcasts via `tokio::sync::broadcast::channel::<()>(1)`** â€” deliberately avoiding `CancellationToken` for a single signal.
- **SOAP fault = HTTP 500 + `<UPnPError>` body** (UPnP convention). Do not leak internal detail to clients; log via `tracing::error` only.
- **DNS rebinding on `/admin/*` is blocked at the `Origin` header**, not via `Host` validation ([http/mod.rs](src/http/mod.rs) `admin_csrf_guard`). When an attacker page rebinds its domain to a LAN IP, the browser still emits `Origin: http://attacker.example.com` â€” the guard rejects any host that isn't a LAN IP / `localhost` / `*.local`. A separate `Host` check would be defense-in-depth only and is intentionally omitted to keep the middleware path simple. If you ever weaken the Origin check, **add a Host check before doing so**.
- **`request_timeout` middleware caps every route at 30s except `/stream/{track_id}`** ([http/mod.rs](src/http/mod.rs)). Stream is exempted because whole-track responses can last tens of minutes; everywhere else a stuck handler must not hold a connection indefinitely. Increase [`REQUEST_TIMEOUT_SECS`](src/http/mod.rs) only after confirming no legitimate Browse/Search/SOAP/admin call comes close to it on the 88k-track baseline.
- **`upnp::xml::{escape_text, escape_attr}` is the single source of XML escaping** ([upnp/xml.rs](src/upnp/xml.rs)). Both helpers are single-pass and drop XML-1.0-disallowed control characters so Linn never sees a parse error. Do not reintroduce per-module `String::replace`-chain escape helpers â€” they're 4-5Ă— slower and (historically) diverged in what they escape.
- **`config.toml` is bootstrap-only; `config_overrides` table is the runtime source of truth** (#13). At startup, the toml values are captured as defaults; user edits go to `config_overrides` (JSON-valued KV), and `AppState.browse` is rebuilt from `catalog::build_browse_settings`. `GET/POST/DELETE /admin/config` drives this. The key catalog lives in `src/config_catalog.rs` â€” add new keys there with a `ReloadTier` (`Runtime` / `Reload` / `Restart`) and a validator.
- **Search dispatches by `upnp:class derivedfrom`** (#5+#10, SPEC Â§5.4). The parser produces `ClassFilter + Predicate` tree (AND / OR / parens / `[@role="..."]`). Album-class returns `alb:{id}` containers, Artist-class returns `aa:{base64}` or `ar:{base64}` containers, Track-class returns track items via 4-field OR. `role="Composer|Conductor|Performer"` routes to the matching `tracks.{composer,conductor,performer}` column at SQL (#9) and switches the Artist-class container kind to `cm:` / `cn:` / `pf:`.
- **Album-class `dc:title` fans out across 3 columns and is ranked** (#21, SPEC Â§5.4). Linn's Album field sends a single-leaf `dc:title contains "X"`; users expect that to find both the artist's own albums and compilations they appear on. `search_albums_ranked` does the 3-way OR (`album_norm` / `effective_album_artist_norm` / `EXISTS tracks.artist_norm`) for WHERE and layers a 4-bucket CASE in ORDER BY: exact album â†’ album_artist contains â†’ partial album â†’ track-artist EXISTS only. Rationale: artist-name queries usually mean "show me this person's records", so the artist bucket beats a partial-title hit like "Beatles Anthology". `upnp:album` predicates stay album-name-only (regression guard).
- **Artist-class search UNIONs album_artist + track artist; `cat:aa` / `cat:ar` are unchanged** (#22, SPEC Â§5.4). `search_artists` runs the predicate against two WHERE clauses (one per source column) via `walk` called twice, composes a `UNION ALL` with `is_aa` flag, then `GROUP BY name` + `MAX(is_aa)` dedupes. Hits where `is_aa = 1` emit `aa:{X}` (parent `cat:aa`); track-only hits emit `ar:{X}` (parent `cat:ar`). Both Browse handlers already exist â€” no new wiring. The two top-level facets keep their role separation: `cat:aa` lists curated album_artists, `cat:ar` lists the noisy union of track artists. Search is the only path that crosses both worlds.
- **`at:{X}` "All tracks by X" shortcut is prepended to `aa:{X}` / `ar:{X}`** (#23). When `tracks.artist_norm = for_search(X)` has â‰Ą 1 row, [`browse::albums`] inserts a synthetic container at index 0 via the shared `shortcut_split` helper; `total_matches` is bumped by 1 and pagination consumes the slot on page 0 (`start=0` â†’ shortcut + albums starting offset 0 with `count-1`; `start>0` â†’ albums starting offset `start-1`). `at:{X}` Browse returns a flat track list across all albums where X is a track-level artist, ordered by album / disc / track. The match is **exact** on the normalized column (`= ?`), not LIKE â€” partial matching is a Search concern.
- **Search fuzzy match goes through `*_norm` shadow columns** (#6, SPEC Â§5.4). `crate::normalize::for_search` runs NFKD â†’ strip marks â†’ lowercase â†’ katakanaâ†’hiragana once per upsert and once on the search input; both sides converge so accents, fullwidth, and halfwidth katakana all match. `db::schema::backfill_search_norms` fills the columns on first migration to v6 via `COALESCE(dst, ?)` â€” manually pinned values survive future upgrades. SearchCriteria's `read_string` slices the source `&str` to preserve multibyte UTF-8 (pre-#6 byte-cast mangled non-ASCII query values).
- **Typo tolerance is a fall-through, not a union** (#28, SPEC Â§5.4). LIKE substring runs first; FTS5 trigram + Jaccard runs *only* when LIKE returned zero AND the normalized query is â‰Ą 3 chars. Mixing both produced tail noise on common queries (`Beatles` returning `Abbey Road` plus trigram-overlap junk like `Atlas Group`). Four pieces: (1) schema v9 adds `albums_fts` / `tracks_fts` FTS5 virtual tables with `tokenize='trigram'` + AFTER triggers, (2) `jaccard_trigram(a, b)` UDF in [src/db/udf.rs](src/db/udf.rs) registered per pooled connection (Rust-native = C-level speed; same UDF in Python would be the bottleneck), (3) `fts5_trigram_or` builds the OR-of-trigrams MATCH expression â€” trigram tokenizer is substring-only, so phrase-matching `"beatlse"` would *not* find `Beatles`; the OR composition is what makes it typo-tolerant, (4) `jaccard_trigram(col, ?) >= 0.2` threshold filters bucket-4 noise. 0.2 is intentionally below pg_trgm's default 0.3 because `beatlse` â†” `the beatles` scores 0.27 (short query against long prefixed name) and would otherwise be cut. Fuzzy applies only to the four single-leaf Contains paths (Linn's Album / Artist / Track / Composer-role fields); compound predicates from non-Linn clients stay LIKE-only.
- **Top-level facets are config-driven** (#8, SPEC Â§6.2). `browse.top_level` (a `Vec<String>` of `cat:*` IDs) chooses both selection and order; default = `config::default_top_level()` (the full canonical list). `categories::root_children` walks the list in order, silently dropping unknown IDs, duplicates after the first occurrence, and classical / year facets when their underlying column is empty. Hi-Res / Lossy / Mixed Quality are surfaced solely by this list â€” there's no separate master switch (an older `browse.quality_categories` toggle was removed once the drag-and-drop reorder widget made it redundant; old `config_overrides` rows for it are now silently ignored). `root_container.childCount` is recomputed from the same pipeline so BrowseMetadata stays consistent with DirectChildren.
- **Year / Decade share `tracks.year`** (#2, SPEC Â§6.2). `tagger::parse_year` accepts "YYYY" / "YYYY-MM-DD" / "(YYYY)" via `ItemKey::Year` falling back to `RecordingDate`; values â‰¤ 0 or â‰Ą 9999 are dropped as sentinels. `cat:yr` enumerates DISTINCT years DESC; `cat:dec` buckets via `(year/10)*10`. Album filters use `WHERE EXISTS (... WHERE t.year = ?)` or `BETWEEN d AND d+9`. Both facets self-hide on libraries with zero populated rows â€” `facet_has_any(ctx, "year")` gates them in `build_root_facet`.
- **ReplayGain stored but not surfaced in DIDL** (#11, SPEC Â§10.2). The four `tracks.rg_*` REAL columns are populated at scan via lofty's `ItemKey::ReplayGain*` variants (Vorbis / TXXX / iTunes mp4 atoms uniformly mapped); `parse_rg` strips an optional `" dB"` suffix and rejects non-finite values. There is no standard UPnP element for gain / peak, so values stay on the row only. `/admin/stats.tracks_with_replaygain` counts rows with `rg_track_gain IS NOT NULL` as the coverage metric â€” track-gain only, not album-gain.
- **`ScanProgress` is a lock-free atomic snapshot** (#12). `scan::run` updates `phase` / `current` / `total` as it moves through walk â†’ tag_read â†’ upsert â†’ postprocess. Tag-read phase spawns a 5-s ticker thread that polls every 500ms (short shutdown latency). `/admin/scan-progress` reads via `Atomic*::load`, no locks.
- **`POST /admin/rescan` is async** (#18). The handler acquires the `scan_lock` permit and returns **202 Accepted** + `{ scan_id, started_at }` immediately; the scan plus post-scan side effects (SystemUpdateID broadcast + random reshuffle) run on a detached `tokio::spawn` whose closure holds the permit until completion. Callers poll `/admin/scan-progress` for progress and read `/admin/scan-report` for the final report. 409 Conflict if a scan is already in flight. Never re-introduce blocking behavior â€” long scans must not hang HTTP clients (especially rsync post-hooks). **`scan_id` is caller-owned**: the rescan handler (and `main` for the startup scan) generates it via `ScanReport::new_id()` and passes it to `scan::run`, so the value in the 202 response IS the value persisted to `last_scan_report` and tagged on every `scan` span log. Previously `scan::run` generated its own id internally and the response carried an unrelated one â€” clients couldn't correlate. Don't revert to internal generation.
- **Startup scan is detached too** (#15). `main` acquires the `scan_lock` permit on the main task (so a closed-semaphore error still aborts startup), then `tokio::spawn`s the scan + post-scan side effects with the same shape as `/admin/rescan`. HTTP `bind` is no longer blocked behind a multi-minute initial scan, so `/admin/scan-progress` is reachable while the very scan it surfaces is running. Shutdown's `scan_lock.acquire().await` still waits for the detached task, so WAL safety on Ctrl-C is unchanged.
- **Companion files are counted, not enumerated** (#19, SPEC Â§4.7 / Â§4.8). The walker has a hard-coded `COMPANION_EXTENSIONS` whitelist ([src/scan/walker.rs](src/scan/walker.rs)) â€” `jpg/jpeg/png/gif/bmp/webp`, `log/cue/nfo/txt/pdf`, `m3u/m3u8/pls`, `md5/sfv/accurip`. Matches increment `WalkResult.companion_files_seen` and flow to `ScanStats.companion_files_seen`; they do NOT push to `skipped`. Non-companion non-audio extensions (stray `.exe`, `.flac.tmp`, mistyped `.mp33`) still go to `skipped` as `unsupported_extension` so they stay actionable. Rationale: on an 88k library, `Folder.jpg` / `.log` files alone produced thousands of skipped entries that drowned out real issues. The list is intentionally not config-driven â€” extend it in code if a new sidecar type proves common in the wild.

## Active pitfalls

- **Gapless playback breaks if Range handling is off** in `/stream/{track_id}`. Both suffix `bytes=-N` and `bytes=N-` must work. Play-count attribution counts only `Range` absent or `start=0`.
- **Compilation albums: trust the compilation flag** (M4A `cpil` / MP3 `TCMP` / Vorbis `COMPILATION`) over Album Artist text. If set, force "Various Artists".
- **Emit `sampleFrequency` / `bitsPerSample` / `nrAudioChannels` correctly** â€” wrong values are why Linn drops 24/192 streams.
- **NAS over SMB/NFS has no btime** â€” fall back to mtime.
- **File move = path change**: DELETE â†’ INSERT, so `added_at` becomes "now". Intentional (SPEC Â§4.2).
- **`MAX(added_at) by album_id` for Recently Added is intentional** â€” a per-album `first_seen_at` would not re-float on new tracks (SPEC Â§6.4).
